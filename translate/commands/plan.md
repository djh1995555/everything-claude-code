# 规划命令

此命令调用 **planner** 代理在编写任何代码之前创建全面的实施计划。

## 此命令的作用

1. **重述需求** - 澄清需要构建什么
2. **识别风险** - 发现潜在问题和阻塞
3. **创建步骤计划** - 将实施分解为阶段
4. **等待确认** - 必须在继续前获得用户批准

## 何时使用

在以下情况使用 `/plan`：
- 开始新功能
- 进行重大架构更改
- 从事复杂重构
- 将影响多个文件/组件
- 需求不清晰或模糊

## 工作原理

规划者代理将：

1. **分析请求** 并以清晰的术语重述需求
2. **分解为阶段** 并提供具体、可操作的步骤
3. **识别依赖项** 在组件之间
4. **评估风险** 和潜在阻塞
5. **估计复杂性**（高/中/低）
6. **呈现计划** 并等待您的明确确认

## 示例用法

```
用户：/plan 我需要添加市场解决时的实时通知

代理（规划者）：
# 实施计划：实时市场解决通知

## 需求重述
- 向正在关注市场的用户发送通知，当市场解决时
- 支持多个通知渠道（应用内、电子邮件、webhook）
- 确保通知可靠传递
- 包含市场结果和用户仓位结果

## 实施阶段

### 阶段 1：数据库模式
- 添加通知表，包含列：id, user_id, market_id, type, status, created_at
- 添加 user_notification_preferences 表用于渠道偏好
- 在 user_id 和 market_id 上创建索引以提高性能

### 阶段 2：通知服务
- 在 lib/notifications.ts 中创建通知服务
- 使用 BullMQ/Redis 实现通知队列
- 添加失败传递的重试逻辑
- 创建通知模板

### 阶段 3：集成点
- 接入市场解决逻辑（当状态更改为"已解决"时）
- 查询在市场中有仓位的所有用户
- 为每个用户排队通知

### 阶段 4：前端组件
- 在标题中创建 NotificationBell 组件
- 添加 NotificationList 模态框
- 通过 Supabase 订阅实现实时更新
- 添加通知偏好页面

## 依赖项
- Redis（用于队列）
- 电子邮件服务（SendGrid/Resend）
- Supabase 实时订阅

## 风险
- 高：电子邮件可传递性（需要 SPF/DKIM）
- 中：每个市场 1000+ 用户的性能
- 中：如果市场经常解决，通知垃圾邮件
- 低：实时订阅开销

## 估计复杂性：中
- 后端：4-6 小时
- 前端：3-4 小时
- 测试：2-3 小时
- 总计：9-13 小时

**等待确认**：继续此计划？（是/否/修改）
```

## 重要说明

**关键**：在您明确用"是"或"继续"或类似的肯定响应确认计划之前，规划者代理**不会**编写任何代码。

如果您想要更改，请用以下响应：
- "修改：[您的更改]"
- "不同方法：[替代方案]"
- "先跳过阶段 2 做阶段 3"

## 与其他命令的集成

规划之后：
- 使用 `/tdd` 通过测试驱动开发实现
- 如果出现构建错误使用 `/build-fix`
- 使用 `/code-review` 审查已完成的实施
