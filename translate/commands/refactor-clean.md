# 重构清理

安全识别和删除死亡代码，每一步都进行测试验证。

## 步骤 1：检测死亡代码

根据项目类型运行分析工具：

| 工具 | 查找内容 | 命令 |
|------|--------------|---------|
| knip | 未使用的导出、文件、依赖项 | `npx knip` |
| depcheck | 未使用的 npm 依赖项 | `npx depcheck` |
| ts-prune | 未使用的 TypeScript 导出 | `npx ts-prune` |
| vulture | 未使用的 Python 代码 | `vulture src/` |
| deadcode | 未使用的 Go 代码 | `deadcode ./...` |
| cargo-udeps | 未使用的 Rust 依赖项 | `cargo +nightly udeps` |

如果没有可用工具，使用 Grep 查找零导入的导出：
```
# 查找导出，然后检查它们是否在任何地方被导入
```

## 步骤 2：分类发现

将发现分类为安全层级：

| 层级 | 示例 | 操作 |
|------|----------|--------|
| **安全** | 未使用的工具、测试辅助函数、内部函数 | 放心删除 |
| **谨慎** | 组件、API 路由、中间件 | 验证没有动态导入或外部消费者 |
| **危险** | 配置文件、入口点、类型定义 | 在接触之前进行调查 |

## 步骤 3：安全删除循环

对于每个安全项目：

1. **运行完整测试套件** — 建立基线（全部绿色）
2. **删除死亡代码** — 使用 Edit 工具进行精确删除
3. **重新运行测试套件** — 验证没有破坏任何东西
4. **如果测试失败** — 立即使用 `git checkout -- <file>` 恢复并跳过此项目
5. **如果测试通过** — 进入下一个项目

## 步骤 4：处理谨慎项目

在删除谨慎项目之前：
- 搜索动态导入：`import()`、`require()`、`__import__`
- 搜索字符串引用：路由名称、配置中的组件名称
- 检查是否从公共包 API 导出
- 验证没有外部消费者（如果已发布，检查依赖者）

## 步骤 5：整合重复项

删除死亡代码后，查找：
- 近似重复的函数（>80% 相似）— 合并为一个
- 冗余类型定义 — 整合
- 没有增加价值的包装函数 — 内联它们
- 没有用途的重新导出 — 删除间接层

## 步骤 6：总结

报告结果：

```
死亡代码清理
──────────────────────────────
已删除：   12 个未使用函数
           3 个未使用文件
           5 个未使用依赖项
已跳过：   2 个项目（测试失败）
已节省：     ~450 行删除
──────────────────────────────
所有测试通过 ✅
```

## 规则

- **删除前务必先运行测试**
- **一次删除一个** — 原子更改使回滚容易
- **如果不确定则跳过** — 保留死亡代码比破坏生产更好
- **清理时不要重构** — 分离关注点（先清理，后重构）
