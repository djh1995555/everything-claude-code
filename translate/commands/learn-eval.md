# /learn-eval - 提取、评估，然后保存

在写入任何技能文件之前，使用质量门和保存位置决策扩展 `/learn`。

## 提取内容

查找：

1. **错误解决模式** — 根本原因 + 修复 + 可重用性
2. **调试技术** — 非显而易见的步骤、工具组合
3. **变通方法** — 库特性、API 限制、版本特定修复
4. **项目特定模式** — 约定、架构决策、集成模式

## 流程

1. 审查会话以查找可提取的模式
2. 识别最有价值/可重复使用的见解

3. **确定保存位置：**
   - 问："这个模式在不同的项目中有用吗？"
   - **全局** (`~/.claude/skills/learned/`)：可在 2+ 个项目中使用的通用模式（bash 兼容性、LLM API 行为、调试技术等）
   - **项目** (当前项目中的 `.claude/skills/learned/`)：项目特定知识（特定配置文件的特性、项目特定的架构决策等）
   - 如有疑问，选择全局（全局 → 项目 比反向更容易）

4. 使用此格式起草技能文件：

```markdown
---
name: pattern-name
description: "130 个字符以内"
user-invocable: false
origin: auto-extracted
---

# [描述性模式名称]

**提取：** [日期]
**上下文：** [此模式适用的简要描述]

## 问题
[这解决什么问题 - 具体说明]

## 解决方案
[模式/技术/变通方法 - 包含代码示例]

## 何时使用
[触发条件]
```

5. **保存前自我评估** 使用此评分标准：

   | 维度 | 1 | 3 | 5 |
   |-----------|---|---|---|
   | 特异性 | 仅抽象原则，无代码示例 | 存在代表性代码示例 | 涵盖所有使用模式的丰富示例 |
   | 可操作性 | 不清楚该做什么 | 主要步骤可理解 | 立即可操作，涵盖边缘情况 |
   | 范围匹配 | 太宽泛或太窄 | 大部分合适，存在一些边界模糊 | 名称、触发器和内容完美对齐 |
   | 非冗余性 | 与另一个技能几乎相同 | 存在一些重叠，但存在独特视角 | 完全独特的价值 |
   | 覆盖率 | 仅涵盖目标任务的一小部分 | 涵盖主要情况，常见变体缺失 | 涵盖主要情况、边缘情况和陷阱 |

   - 每个维度评分 1-5
   - 如果任何维度评分为 1-2，改进草稿并重新评分，直到所有维度 ≥ 3
   - 向用户显示评分表和最终草稿

6. 请求用户确认：
   - 显示：建议的保存路径 + 评分表 + 最终草稿
   - 在写入前等待明确确认

7. 保存到确定的位置

## 步骤 5 的输出格式（评分表）

| 维度 | 分数 | 理由 |
|-----------|-------|-----------|
| 特异性 | N/5 | ... |
| 可操作性 | N/5 | ... |
| 范围匹配 | N/5 | ... |
| 非冗余性 | N/5 | ... |
| 覆盖率 | N/5 | ... |
| **总计** | **N/25** | |

## 注释

- 不要提取简单的修复（拼写错误、简单语法错误）
- 不要提取一次性问题（特定的 API 中断等）
- 专注于能在未来会话中节省时间的模式
- 保持技能聚焦 — 每个技能一个模式
- 如果覆盖率分数低，在保存前添加相关变体
