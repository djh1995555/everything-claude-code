# Ralphinho RFC 管道

灵感来自 [humanplane](https://github.com/humanplane) 风格的 RFC 分解模式和多单元编排工作流。

当功能太大无法由单个代理完成，必须拆分为可独立验证的工作单元时使用此技能。

## 管道阶段

1. RFC 接收
2. DAG 分解
3. 单元分配
4. 单元实现
5. 单元验证
6. 合并队列和集成
7. 最终系统验证

## 单元规范模板

每个工作单元应包括：
- `id`
- `depends_on`
- `scope`
- `acceptance_tests`
- `risk_level`
- `rollback_plan`

## 复杂度层级

- 第 1 层：孤立文件编辑，确定性测试
- 第 2 层：多文件行为变更，中等集成风险
- 第 3 层：架构/认证/性能/安全变更

## 每个单元的质量管道

1. 研究
2. 实现计划
3. 实现
4. 测试
5. 审查
6. 合并就绪报告

## 合并队列规则

- 永远不要合并存在未解决依赖失败的单元。
- 始终将单元分支变基到最新的集成分支。
- 每次队列合并后重新运行集成测试。

## 恢复

如果单元停滞：
- 从活动队列中驱逐
- 快照发现
- 重新生成缩小的单元范围
- 使用更新的约束重试

## 输出

- RFC 执行日志
- 单元记分卡
- 依赖关系图快照
- 集成风险总结
