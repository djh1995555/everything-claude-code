# /search-first — 编码前先研究

将“在实现前搜索现有解决方案”的工作流系统化。

## 触发条件

在以下情况下使用此技能：
- 启动一个可能有现有解决方案的新功能
- 添加依赖或集成
- 用户要求“添加 X 功能”而你即将编写代码
- 在创建新的实用工具、助手或抽象之前

## 工作流

```
┌─────────────────────────────────────────────┐
│  1. 需求分析                               │
│     定义所需功能                             │
│     识别语言/框架约束                       │
├─────────────────────────────────────────────┤
│  2. 并行搜索（研究员代理）                  │
│     ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│     │  npm /   │ │  MCP /   │ │  GitHub / │  │
│     │  PyPI    │ │  技能    │ │  网络    │  │
│     └──────────┘ └──────────┘ └──────────┘  │
├─────────────────────────────────────────────┤
│  3. 评估                                   │
│     对候选方案评分（功能、维护、            │
│     社区、文档、许可证、依赖）              │
├─────────────────────────────────────────────┤
│  4. 决策                                   │
│     ┌─────────┐  ┌──────────┐  ┌─────────┐  │
│     │  采用   │  │  扩展    │  │  构建    │  │
│     │  原样使用│  │  /包装   │  │  自定义  │  │
│     └─────────┘  └──────────┘  └─────────┘  │
├─────────────────────────────────────────────┤
│  5. 实现                                   │
│     安装包 / 配置 MCP /                     │
│     编写最少的自定义代码                     │
└─────────────────────────────────────────────┘
```

## 决策矩阵

| 信号 | 行动 |
|--------|--------|
| 完全匹配、维护良好、MIT/Apache 许可证 | **采用** — 直接安装使用 |
| 部分匹配、良好基础 | **扩展** — 安装 + 编写薄包装 |
| 多个弱匹配 | **组合** — 结合 2-3 个小包 |
| 未找到合适方案 | **构建** — 编写自定义代码，但基于研究结果 |

## 使用方法

### 快速模式（内联）

在编写实用工具或添加功能之前，在脑海中运行以下步骤：

0. 此功能是否已存在于 repo 中？→ 首先使用 `rg` 搜索相关模块/测试
1. 这是常见问题吗？→ 搜索 npm/PyPI
2. 是否有相关的 MCP？→ 检查 `~/.claude/settings.json` 并搜索
3. 是否有相关的技能？→ 检查 `~/.claude/skills/`
4. 是否有 GitHub 实现/模板？→ 在编写全新代码之前，搜索维护良好的 OSS

### 完整模式（代理）

对于非 trivial 功能，启动研究员代理：

```
Task(subagent_type="general-purpose", prompt="
  研究现有工具：[描述]
  语言/框架：[语言]
  约束：[任何]

  搜索：npm/PyPI、MCP 服务器、Claude Code 技能、GitHub
  返回：结构化比较和推荐
")
```

## 按类别划分的搜索快捷方式

### 开发工具
- 代码检查 → `eslint`、`ruff`、`textlint`、`markdownlint`
- 格式化 → `prettier`、`black`、`gofmt`
- 测试 → `jest`、`pytest`、`go test`
- 预提交 → `husky`、`lint-staged`、`pre-commit`

### AI/LLM 集成
- Claude SDK → Context7 获取最新文档
- 提示管理 → 检查 MCP 服务器
- 文档处理 → `unstructured`、`pdfplumber`、`mammoth`

### 数据与 APIs
- HTTP 客户端 → `httpx`（Python）、`ky`/`got`（Node）
- 验证 → `zod`（TS）、`pydantic`（Python）
- 数据库 → 首先检查 MCP 服务器

### 内容与发布
- Markdown 处理 → `remark`、`unified`、`markdown-it`
- 图像优化 → `sharp`、`imagemin`

## 集成点

### 与规划器代理

规划器应在第 1 阶段（架构审查）之前调用研究员：
- 研究员识别可用工具
- 规划器将它们纳入实现计划
- 避免在计划中“重新发明轮子”

### 与架构师代理

架构师应向研究员咨询：
- 技术栈决策
- 集成模式发现
- 现有参考架构

### 与 iterative-retrieval 技能

结合使用进行渐进式发现：
- 周期 1：广泛搜索（npm、PyPI、MCP）
- 周期 2：详细评估顶级候选方案
- 周期 3：测试与项目约束的兼容性

## 示例

### 示例 1：“添加死链接检查”

```
需求：检查 markdown 文件中的断链
搜索：npm "markdown dead link checker"
发现：textlint-rule-no-dead-link（评分：9/10）
行动：采用 — npm install textlint-rule-no-dead-link
结果：零自定义代码，经过实战测试的解决方案
```

### 示例 2：“添加 HTTP 客户端包装”

```
需求：具有重试和超时处理的弹性 HTTP 客户端
搜索：npm "http client retry"、PyPI "httpx retry"
发现：got（Node）带重试插件、httpx（Python）带内置重试
行动：采用 — 直接使用 got/httpx 并配置重试
结果：零自定义代码，经过生产验证的库
```

### 示例 3：“添加配置文件检查器”

```
需求：根据模式验证项目配置文件
搜索：npm "config linter schema"、"json schema validator cli"
发现：ajv-cli（评分：8/10）
行动：采用 + 扩展 — 安装 ajv-cli，编写项目特定模式
结果：1 个包 + 1 个模式文件，无自定义验证逻辑
```

## 反模式

- **直接编码**：在未检查是否存在的情况下编写实用工具
- **忽略 MCP**：未检查 MCP 服务器是否已提供该功能
- **过度定制**：过度包装库以至于失去其优势
- **依赖膨胀**：为一个小功能安装大型包
