# 代理 harness 构建

当您需要改进代理的规划、工具调用、错误恢复和完成收敛方式时，使用此技能。

## 核心模型

代理输出质量受以下因素限制：
1. 操作空间质量
2. 观察质量
3. 恢复质量
4. 上下文预算质量

## 操作空间设计

1. 使用稳定、明确的工具名称。
2. 保持输入以模式为先且范围狭窄。
3. 返回确定性的输出形状。
4. 除非隔离不可能，否则避免使用万能工具。

## 粒度规则

- 对高风险操作（部署、迁移、权限）使用微型工具。
- 对常见的编辑/读取/搜索循环使用中型工具。
- 仅当往返开销是主要成本时才使用宏工具。

## 观察设计

每个工具响应应包含：
- `status`: success|warning|error
- `summary`: 一行结果
- `next_actions`: 可操作的后续步骤
- `artifacts`: 文件路径/ID

## 错误恢复契约

对于每个错误路径，应包含：
- 根本原因提示
- 安全重试说明
- 明确的停止条件

## 上下文预算

1. 保持系统提示最小化且不变。
2. 将大型指南移至按需加载的技能中。
3. 优先引用文件而非内联长文档。
4. 在阶段边界处压缩，而非任意令牌阈值。

## 架构模式指导

- ReAct：最适合路径不确定的探索性任务。
- 函数调用：最适合结构化的确定性流程。
- 混合（推荐）：ReAct规划 + 类型化工具执行。

## 基准测试

跟踪：
- 完成率
- 每个任务的重试次数
- pass@1 和 pass@3
- 每个成功任务的成本

## 反模式

- 太多语义重叠的工具。
- 不透明的工具输出，没有恢复提示。
- 仅错误输出，没有下一步。
- 上下文过载，包含无关引用。
